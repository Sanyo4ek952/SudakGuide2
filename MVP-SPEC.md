# MVP-SPEC.md — Сервис бронирования жилья (Судак)

## 0) Коротко о продукте

MVP веб‑сервиса для поиска и бронирования жилья в городе Судак. Пользователь без регистрации может смотреть опубликованные объекты и рассчитывать стоимость по датам. Для создания заявки на бронирование нужна регистрация. Оплата **не входит** в MVP.

Ключевой принцип MVP: **заявка на бронь = временная блокировка дат (hold)**, чтобы исключить двойное бронирование.

---

## 1) Цели MVP

1. Дать гостю (без логина) каталог опубликованных объектов и карточку объекта с расчётом цены по выбранным датам.
2. Дать пользователю регистрацию/логин и возможность отправить заявку на бронирование.
3. Дать проверенному хозяину (HOST) возможность добавлять и управлять объектами и ценами.
4. Дать супер‑админу (ADMIN) модерацию объектов/хозяев и базовую аналитику + логи/ошибки.
5. Уведомлять хозяина о новой заявке через **Telegram (только уведомление + ссылка)**.

---

## 2) Не цели (не делаем в MVP)

* Онлайн‑оплата и интеграция платёжных систем.
* Подтверждение/отклонение брони через Telegram‑бот (кнопки). Только подтверждение на сайте.
* Интеграции календарей (iCal/Google Calendar), синхронизации с внешними площадками.
* Карта/координаты/геокодинг.
* Отзывы, рейтинги.
* Импорт/экспорт CSV/Excel.
* Мультиязычность (только RU).

---

## 3) Роли и права

### 3.1 Guest (неавторизованный)

* Видит **только PUBLISHED** объекты.
* Может открывать карточку объекта.
* Может выбирать даты и видеть расчёт итоговой суммы.
* Не может отправлять заявку на бронь.

### 3.2 User (обычный зарегистрированный)

* Всё, что Guest.
* Может создавать заявку на бронирование (BookingRequest) на опубликованный объект.
* Может смотреть свои заявки и их статусы.

### 3.3 Host (проверенный хозяин)

> Роль выдаётся **только ADMIN’ом**.

* Может создавать/редактировать свои объекты (Listing) и управлять контентом.
* Может добавлять/редактировать фото.
* Может задавать ценовые периоды (PricePeriod).
* Может видеть заявки на бронирование по своим объектам.
* Может подтверждать/отклонять заявки.

### 3.4 Admin (супер‑админ)

* Управляет пользователями: назначает роль HOST.
* Модерирует объекты: переводит из PENDING_REVIEW в PUBLISHED или REJECTED.
* Имеет доступ к базовой аналитике по воронке и событиям.
* Имеет доступ к логам/ошибкам.

## 4) Основные сущности (домены)

### 4.1 Listing (объект жилья)

**Обязательные поля (MVP):**

* title (название)
* address (адрес)
* district (район)
* guests (вместимость)
* rooms (кол-во комнат)
* beds (кол-во спальных мест/кроватей)
* amenities (удобства)
* rules (правила проживания)
* checkInTime (время заезда)
* checkOutTime (время выезда)
* photos (минимум 1 фото для публикации)

**Статусы Listing:**

* DRAFT — черновик
* PENDING_REVIEW — отправлен на модерацию
* PUBLISHED — опубликован
* REJECTED — отклонён

### 4.2 PricePeriod (периоды цен)

* listingId
* dateFrom (включительно)
* dateTo (исключительно)
* pricePerNight (цена за ночь)

**Правило расчёта:**

* Итог = сумма по ночам между dateFrom/dateTo заявки.
* Если для ночи нет периода цены → это ошибка в данных (объект считается недоступным для расчёта на эти даты).

### 4.3 BookingRequest (заявка на бронь)

* listingId
* userId
* dateFrom, dateTo
* nights
* totalPrice
* status
* expiresAt (момент истечения hold)

**Статусы BookingRequest:**

* PENDING — создана пользователем, даты заблокированы до expiresAt
* APPROVED — подтверждено хозяином, даты считаются забронированными
* REJECTED — отклонено хозяином, даты освобождаются
* EXPIRED — время истекло, даты освобождаются
* CANCELED — отменено пользователем (если включим в MVP — опционально)

---

## 5) Ключевые бизнес‑правила

### 5.1 Доступность и анти‑овербук

* При создании заявки система обязана проверить отсутствие пересечений по датам с:

  * активными PENDING заявками (expiresAt > now)
  * APPROVED заявками
* Если пересечение найдено → заявка не создаётся (конфликт дат).

### 5.2 Hold при заявке

* Создание заявки **сразу блокирует даты**.
* expiresAt задаётся глобальной настройкой (например, 2 часа) и может быть изменён админом.
* По истечении expiresAt заявка автоматически становится EXPIRED и перестаёт блокировать даты.

### 5.3 Публикация объектов

* В каталоге и карточке доступны только Listings со статусом PUBLISHED.
* Host может редактировать объект в любом статусе, но публикация — только после модерации.

### 5.4 Роли

* Обычный пользователь не может создавать Listing.
* Host роль выдаёт только Admin.

## 6) Основные пользовательские флоу (end‑to‑end)

### 6.1 Просмотр каталога (Guest)

1. Открыть главную/каталог.
2. Найти объект через поиск/фильтры.
3. Открыть карточку.
4. Выбрать даты → увидеть итоговую сумму.

### 6.2 Регистрация и создание заявки (User)

1. Зарегистрироваться/войти.
2. Открыть карточку опубликованного объекта.
3. Выбрать даты.
4. Отправить заявку на бронь.
5. Получить статус PENDING и время истечения.

### 6.3 Подтверждение заявки (Host)

1. Host получает Telegram‑уведомление о новой заявке.
2. Переходит по ссылке на страницу заявки.
3. Подтверждает или отклоняет.
4. Пользователь видит финальный статус.

### 6.4 Создание объекта и модерация (Host → Admin)

1. Admin назначает пользователю роль HOST.
2. Host создаёт объект, заполняет поля, загружает фото.
3. Host отправляет на модерацию → PENDING_REVIEW.
4. Admin публикует → PUBLISHED или отклоняет → REJECTED.

## 7) Страницы и минимальный UI

### Публичные

* `/` или `/listings` — каталог (поиск/фильтры/сортировки)
* `/listings/[id]` — карточка объекта (галерея, описание, календарь, расчёт суммы)
* `/auth/sign-in`, `/auth/sign-up` — вход/регистрация

### Личный кабинет пользователя

* `/account/bookings` — мои заявки/брони

### Кабинет хозяина (HOST)

* `/host/listings` — мои объекты
* `/host/listings/new` — создание объекта
* `/host/listings/[id]/edit` — редактирование
* `/host/listings/[id]/prices` — управление периодами цен
* `/host/bookings` — заявки по моим объектам
* `/host/bookings/[id]` — карточка заявки, approve/reject

### Админка (ADMIN)

* `/admin/users` — список пользователей, назначение роли HOST
* `/admin/listings` — модерация объектов
* `/admin/analytics` — базовые метрики
* `/admin/logs` — просмотр ошибок/логов (минимально)

---

## 8) Фильтры/поиск/сортировки (каталог)

**MVP:**

* Поиск по названию (title).
* Фильтр по цене (min/max) — по итоговой цене за выбранный период или по цене/ночь (уточнить реализацию; по умолчанию: по цене/ночь на первую ночь выбранного периода).
* Фильтр по комнатам (rooms).
* Фильтр по датам: показывать объекты, доступные на выбранный период.
* Сортировки (минимум): по цене (возр/убыв), по новизне.

---

## 9) Telegram уведомления

* При создании BookingRequest (PENDING) отправлять сообщение хозяину (owner) в Telegram.
* Сообщение содержит:

  * объект
  * даты
  * сумма
  * ссылку на страницу заявки на сайте
* В MVP Telegram используется **только как канал уведомления**, без интерактивных кнопок.

## 11) Definition of Done (DoD) для MVP

MVP считается готовым, когда:

1. Каталог и карточка объекта работают для гостя (без логина) и показывают только PUBLISHED.
2. Пользователь может зарегистрироваться и создать заявку на бронь.
3. Даты блокируются сразу при создании заявки и не допускают пересечений.
4. Host может подтвердить/отклонить заявку на сайте.
5. Истечение hold переводит заявку в EXPIRED и освобождает даты.
6. Host может создать объект и отправить на модерацию.
7. Admin может назначать HOST и модерировать объекты.
8. Telegram уведомление хозяину приходит при новой заявке.
9. Все gates проходят: `pnpm check`, `pnpm build`.
10. Пройден e2e smoke по основным флоу (регистрация → заявка, host approve/reject, модерация публикации).
